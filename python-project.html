03 — Task 1.1: Database Setup

setup_database.py —
Automated DB Creation

The setup script automates the complete database initialization pipeline in a single command. It is designed to be idempotent — running it multiple times produces the same clean result, because it deletes any existing database file before recreating it (solving the UNIQUE constraint re-run problem). The CSV importer is generic — it reads the header row dynamically and builds parameterized INSERT statements, making it reusable for any table without modification.

1
Database Teardown & Recreation
Checks for existing data/myproject.db and removes it with os.remove() before creating fresh. Ensures idempotency across reruns — no UNIQUE constraint failures on customer_id or email.
2
Schema DDL — 3 Tables with Full Constraints
Creates customers (UNIQUE email), products (CHECK price >= 0, CHECK cost >= 0), and orders (FK → customers, FK → products, CHECK quantity > 0). All via PRAGMA foreign_keys = ON.
3
Generic CSV Importer
Reads header row dynamically. Builds INSERT statement as INSERT INTO {table} ({columns}) VALUES ({placeholders}) — parameterized for SQL injection safety. Replaces empty strings with NULL. Bulk-inserts via executemany() for performance.
4
Verification — Row Counts Printed
Runs SELECT COUNT(*) on each table and prints results — confirming customers: 30, products: 20, orders: 100 after every run.
04 — Task 1.2: Data Analysis

analyze_data.py —
5 Queries, Dual Implementation

The centerpiece of the project. Queries 1 and 2 use pure SQL. Queries 3, 4, and 5 each have two complete implementations — an SQL version using GROUP BY and aggregation functions, and a Python version that fetches raw joined rows and performs all grouping, aggregation, and sorting in Python using dictionaries, loops, and sorted(). Both versions return identical results — verified by visual comparison during development.

Q1 — Customers by CitySQL Only
Business question: Which customers are in Boston for local marketing campaigns? Parameterized with ? — safe against SQL injection, reusable for any city.
SELECT · WHERE · parameterized query · ORDER BY name
SELECT customer_id, name, email, city, join_date FROM customers WHERE city = ? -- parameterized, prevents SQL injection ORDER BY name
Q2 — Revenue by Product CategorySQL Only
Business question: Which product categories generate the most revenue? Reveals that Electronics drives $3,506 of the $4,651 total — 75% of all revenue — a key portfolio concentration insight.
JOIN (orders × products) · SUM(quantity × price) · GROUP BY category · ORDER BY revenue DESC
SELECT p.category, SUM(o.quantity * p.price) AS total_revenue FROM orders o JOIN products p ON o.product_id = p.product_id GROUP BY p.category ORDER BY total_revenue DESC
Q3 — Top Customers by Total SpendingSQL + Python
Business question: Who are the top 5 most valuable customers by lifetime revenue? Critical for loyalty programs, VIP tiers, and account-based marketing.
SQL: 3-table JOIN · SUM · GROUP BY · LIMIT | Python: dict aggregation · lambda sort · list slicing
SQL Version
SELECT c.customer_id, c.name, SUM(o.quantity * p.price) AS total_spent FROM customers c JOIN orders o ON c.customer_id = o.customer_id JOIN products p ON o.product_id = p.product_id GROUP BY c.customer_id, c.name ORDER BY total_spent DESC LIMIT ?
Python Version (no GROUP BY / SUM in SQL)
# Pull raw rows — no aggregation in SQL totals = {} for cid, name, qty, price in rows: revenue = qty * price if cid not in totals: totals[cid] = {"name": name, "total": 0.0} totals[cid]["total"] += revenue result = [(cid, d["name"], d["total"]) for cid, d in totals.items()] result.sort(key=lambda x: x[2], reverse=True) return result[:limit]
Q4 — Monthly Sales TrendSQL + Python
Business question: How is revenue trending month by month? Enables seasonality detection, forecasting, and marketing calendar planning.
SQL: strftime('%Y-%m') date grouping (SQLite-specific) | Python: order_date[:7] string slicing · dict accumulation · sorted() by key
SQL Version — strftime() for SQLite
SELECT strftime('%Y-%m', o.order_date) AS year_month, SUM(o.quantity * p.price) AS total_revenue FROM orders o JOIN products p ON o.product_id = p.product_id GROUP BY year_month ORDER BY year_month
Python Version — string slicing for month grouping
monthly = {} for order_date, qty, price in rows: # 'YYYY-MM-DD'[:7] → 'YYYY-MM' ym = order_date[:7] revenue = qty * price if ym not in monthly: monthly[ym] = 0.0 monthly[ym] += revenue return sorted(monthly.items(), key=lambda x: x[0])
Q5 — Product Profitability (Revenue − Cost)SQL + Python
Business question: Which products generate the highest total profit (not just revenue)? Headphones lead at $359.92 profit — despite lower sales volume than Wireless Mouse, higher margin wins. Electronics category dominates: top 5 products all Electronics.
SQL: SUM(qty × (price−cost)) profit formula | Python: per-product dict with revenue + cost accumulation, profit computed post-aggregation
-- SQL version: profit = revenue - cost at line item level, aggregated per product SELECT p.name, SUM(o.quantity * p.price) AS total_revenue, SUM(o.quantity * p.cost) AS total_cost, SUM(o.quantity * (p.price - p.cost)) AS total_profit FROM products p JOIN orders o ON p.product_id = o.product_id GROUP BY p.product_id, p.name ORDER BY total_profit DESC LIMIT ?

07 — Technical Challenges

Three Real Problems
Solved from First Principles

C1
Idempotent Database Setup — UNIQUE Constraint Failures
Problem: Rerunning setup_database.py on an existing database caused sqlite3.IntegrityError: UNIQUE constraint failed: customers.customer_id — because the same rows were being inserted into tables that already had those primary keys.

Solution: Added os.path.exists(db_path) check + os.remove(db_path) at the start of create_database() — the database is always recreated clean. This makes the script idempotent: running it 10 times produces the same result as running it once.
C2
Matching SQL and Python Query Results Exactly
Problem: For Q3–Q5, both implementations had to return identical results — same totals, same ordering. Dictionary-based aggregation in Python could produce floating-point rounding differences or ordering ties handled differently than SQL's ORDER BY.

Solution: Written the SQL version first, verified numbers looked correct, then built the Python version to match — using result.sort(key=lambda x: x[2], reverse=True) to replicate ORDER BY DESC. Ran both versions side-by-side in main() and compared every row visually. Confirmed match for all three queries.
C3
SQLite Date Functions vs. MySQL Syntax
Problem: Initial AI suggestion for monthly grouping used MONTH(order_date) — valid in MySQL/PostgreSQL but not in SQLite, which has no MONTH() function. This caused a syntax error immediately on execution.

Solution: Researched SQLite date functions and replaced with strftime('%Y-%m', order_date) — SQLite's native date string formatter. Mirrored this in the Python version using order_date[:7] string slicing — elegant parity between the two implementations.

The pipeline answers every operational question an e-commerce analytics team would need weekly: which customers to target by city (Q1), which product categories drive revenue (Q2 — Electronics: 75%), who the top 5 customers are (Q3), how sales are trending month-by-month (Q4), and which products are most profitable after cost (Q5 — Headphones leads at $360 profit). All outputs are programmatically generated and repeatable — run the pipeline on fresh data and the reports update automatically.

The dual SQL/Python implementation demonstrates something more important than just two ways to get the same answer: it shows when to push work to the database vs. when to handle it in application code — a decision every data engineer and analytics engineer makes daily. SQL is optimal for aggregation at scale; Python is necessary when business logic is too complex for SQL, when you need dynamic conditional logic, or when you're building reusable functions for a larger application.
