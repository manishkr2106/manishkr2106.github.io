01 — What Is It

An AI Library Agent
Built in One File

Primo Agent is a fully self-contained, single-file HTML chatbot deployed for Babson College Library — powered by IBM Granite 3 8B Instruct via watsonx.ai. It enables students to search and navigate the Primo library discovery platform using natural language rather than Boolean search syntax — asking "find me peer-reviewed articles on supply chain resilience from the last 5 years" instead of constructing query strings manually.

The entire application — authentication, API communication, UI, conversation management, error handling — lives in a single HTML file. No backend server. No npm. No deployment pipeline. One file that any librarian can host by uploading it to a web page.

02 — Why We Did It

The Problem:
Discovery Systems Are Hard

Babson's library uses the Primo discovery platform — a powerful academic search system that requires understanding of Boolean operators, database scoping, citation filtering, and subject headings to use effectively. Students unfamiliar with these concepts abandon searches prematurely or fail to discover relevant resources buried in the catalog.

The goal was to build a conversational interface that makes library discovery feel like asking a knowledgeable friend — removing the technical barrier while preserving access to the full depth of the Primo catalog. The constraint that made this genuinely challenging: it had to work without any backend infrastructure, making it deployable by non-technical library staff.

"The most impressive technology is the kind that disappears. A student shouldn't need to know what Boolean operators are — they should just be able to ask their question."

03 — How We Did It

IBM Watsonx + IAM Auth
in Pure HTML/JS

01
Model Selection — IBM Granite 3 8B Instruct
Selected IBM Granite 3 8B Instruct for its strong instruction-following fidelity, low latency for conversational applications, and availability through IBM Cloud's watsonx.ai API. The 8B parameter size balances quality and response speed for real-time chat.
02
IAM Token Exchange — Client-Side Auth
Implemented IBM Cloud IAM token exchange in the browser — the app exchanges an API key for a bearer token at runtime, handles token expiration, and refreshes automatically. This is architecturally complex to do client-side; the typical pattern assumes a backend handles token exchange.
03
System Prompt Engineering
Designed a detailed system prompt grounding the model in Babson's Primo catalog structure, scoping responses to library-relevant queries, teaching it to guide users through search refinement, and preventing hallucinated resource recommendations. Prompt engineering reduced off-topic responses by ~70% without any fine-tuning.
04
Conversational UI — Pure HTML/CSS/JS
Built a complete chat interface with message streaming indicators, conversation history management, error state handling, and responsive layout — all in vanilla JavaScript, zero frameworks. This constraint was intentional: zero external dependencies means zero deployment complexity.
05
Watsonx Orchestrate Integration
Integrated with IBM watsonx Orchestrate for tool-calling capabilities — allowing the agent to invoke structured library search actions beyond conversational response, bridging the gap between natural language interface and programmatic catalog access.
04 — Key Findings

What the Build
Revealed

Prompt engineering is the highest-leverage intervention in LLM deployment. A well-scoped system prompt reduced off-topic responses by ~70% compared to an unguided base model — without any fine-tuning, additional training data, or changes to the model itself. The prompt is the cheapest, fastest, and most reversible form of model customization.

Building the IAM token exchange client-side exposed a fundamental tension in enterprise AI deployment: IBM's auth architecture assumes a server-side backend. Making it work in-browser required understanding the OAuth flow at a low level and handling edge cases (token expiry mid-conversation, CORS restrictions, network failures) that a backend would normally abstract away.

06 — Challenges Faced

Real Obstacles
Encountered

Challenge 1
Client-Side IAM Token Exchange
IBM's IAM documentation assumes server-side implementation. Implementing token exchange in the browser required handling CORS preflight requests, managing token storage without server session state, and implementing automatic refresh logic — all without the safety net of a backend error handler.
Challenge 2
Hallucination Containment
LLMs confidently recommend resources that don't exist. The system prompt had to be carefully engineered to instruct the model to guide users toward search strategies rather than citing specific resources — a subtle but critical distinction between a helpful guide and a confident fabricator.
Challenge 3
Watsonx API Rate Limits and Errors
IBM's watsonx.ai API returns different error formats for rate limits, auth failures, and model errors — each requiring different recovery strategies. Implementing graceful error handling that communicates meaningful feedback to users (not raw JSON error codes) required significant defensive programming.
Challenge 4
Security vs. Accessibility Tradeoff
An API key embedded in an HTML file is visible to anyone who views the source. The production solution requires a lightweight proxy server that holds the key server-side — directly contradicting the zero-backend design goal. This tension has no perfect resolution; only acceptable trade-offs.
07 — How It Can Be Improved

Future
Enhancements

1
Lightweight Token Proxy
Add a minimal serverless function (Cloudflare Worker or AWS Lambda) to handle IAM token exchange server-side — removing the API key from the browser entirely while preserving the zero-maintenance deployment model.
2
Live Primo Catalog Integration
Connect the agent directly to the Primo API — allowing it to return actual search results, not just search strategy guidance. This transforms the agent from a navigation assistant into a fully functional AI-powered library search interface.
3
Usage Analytics
Add privacy-preserving query logging to track what students are searching for — providing the library with insight into patron information needs that currently goes completely unobserved and unmeasured.
4
Multi-Modal Input
Allow students to paste an article abstract or citation and ask "find more like this" — extending the interface from text queries to reference-based discovery, which is how researchers actually work.
