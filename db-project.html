03 — Phase 2: Data Generation

Mockaroo —
Realistic Test Data

Phase 2Mockaroo → 9 UTF-8 CSV files
One Mockaroo schema was built per entity with field names matching the SQLite schema exactly. Data ranges were constrained realistically: budgets $38K–$495K, dates between 2023-01-01 and 2025-12-31, allocation percentages 0–100, estimated task hours 0–90. Boolean flags (is_active, completed) used realistic True/False distributions. Enum fields (status, category, role) used custom lists matching the scenario. Parent tables were generated before child tables — enforcing the same dependency order as the eventual import sequence.
01
Parent Tables First (no FK dependencies)
Generated in order: clients → project_stages → priority_levels → expense_categories → people. All IDs in these tables start at 1 and increment — so child tables can reference valid ranges (e.g., client_id 1–30, person_id 1–40).
02
Child Tables Second (reference parent ID ranges)
projects (client_id: 1–30, stage_id: 1–7) → tasks (project_id: 1–50, assigned_to: 1–40, priority_id: 1–4) → expenses (project_id: 1–50, expense_category_id: 1–9) → project_members (project_id: 1–50, person_id: 1–40). Every FK field was constrained to the parent's exact ID range.
03
Data Cleaning in Excel Before Import
Each CSV reviewed for UTF-8 encoding, correct headers, absence of NULL primary keys. The project_members CSV required manual deduplication — Mockaroo generated duplicate (project_id, person_id) pairs that violated the UNIQUE constraint. These were caught and removed in Excel before import.

04 — Phase 3: Database Implementation

SQLite Schema &
DBeaver Import

Phase 3schema.sql → DBeaver → Validated Database
The full DDL was hand-written in DBeaver's SQL editor with PRAGMA foreign_keys = ON; at the top — enforcing referential integrity at the database engine level, not just by convention. Every CREATE TABLE statement was written manually, tested, and verified before CSV import began. Auto-generated DDL from AI was cross-checked line by line against the ERD before use.
3.1
SQLite Database File Created in DBeaver
New SQLite connection created via Database → New Database Connection → SQLite. File path set, test connection verified, PRAGMA foreign_keys = ON confirmed active for the session.
3.2
DDL Written and Executed — All 9 Tables
Every CREATE TABLE statement written in the SQL editor with correct SQLite types (INTEGER for IDs and booleans, REAL for money/hours, TEXT for strings and dates — SQLite has no native DATE type). All PK/FK/UNIQUE/CHECK/DEFAULT constraints included. Each table executed individually and verified in the Database Navigator before the next was created.
3.3
CSV Import — Parent-to-Child Order
Imported using DBeaver's Import Data wizard with manual column mapping for every table (auto-mapping rejected after it produced silent mismatches like assigned_person vs. assigned_to). Import order: clients → project_stages → priority_levels → expense_categories → people → projects → tasks → expenses → project_members. Each table verified with SELECT COUNT(*) after import.
3.4
Data Quality Validation — Zero Integrity Errors
Ran the full suite of quality checks: orphan FK checks (expenses → projects, tasks → projects, tasks → people, project_members → both), NULL checks on required fields, value-range checks (amount <= 0, estimated_hours < 0 or > 200, allocation_pct > 100), and aggregate sanity checks (total spend vs. budget per project, task completion rate per project). All checks returned zero violations.
05 — Phase 4: Business Intelligence Queries

8 SQL Queries —
Real Operational Answers

Eight queries were written to answer real questions a BabsonAI Labs CFO, project manager, or team lead would ask weekly. Six address the required business questions for Scenario A; two are original queries designed to surface insights the required set doesn't cover. All use CTEs, LEFT JOINs with COALESCE for null-safe math, NULLIF for division-by-zero safety, and SQLite-specific date parsing via substr() — since SQLite stores dates as TEXT with no native date type.

Q1 — Budget vs. Spend VarianceRequired
Business question: Which projects are over budget, and by how much? Sorted by largest overrun — triggers cost control or scope review before the situation worsens.
Techniques: CTE · LEFT JOIN · COALESCE · NULLIF (division-by-zero) · ROUND · ORDER BY computed column
WITH spend AS ( SELECT project_id, SUM(amount) AS total_spend FROM expenses GROUP BY project_id ) SELECT p.project_name, ROUND(p.budget, 2) AS budget, ROUND(COALESCE(s.total_spend, 0), 2) AS spend_to_date, ROUND(COALESCE(s.total_spend, 0) - p.budget, 2) AS variance_amount, ROUND(100.0 * COALESCE(s.total_spend,0) / NULLIF(p.budget,0), 1) AS pct_of_budget FROM projects p LEFT JOIN spend s USING(project_id) ORDER BY variance_amount DESC;
Q2 — Workload per Person (Last 30 Days)Required
Business question: For each team member, how many tasks were completed and how many are pending within the last 30 days? Enables workload rebalancing before deadlines slip.
Techniques: CTE with date params · conditional SUM(CASE WHEN) · substr() date parsing (SQLite TEXT dates) · LEFT JOIN · GROUP BY
WITH params AS ( SELECT date('now','-30 day') AS d30, date('now') AS today ) SELECT pe.first_name || ' ' || pe.last_name AS person_name, SUM(CASE WHEN t.completed = 1 AND date(substr(t.completion_date,7,4)||'-'||substr(t.completion_date,1,2)||'-'||substr(t.completion_date,4,2)) BETWEEN (SELECT d30 FROM params) AND (SELECT today FROM params) THEN 1 ELSE 0 END) AS completed_30d, SUM(CASE WHEN t.completed = 0 AND date(substr(t.due_date,7,4)||'-'||substr(t.due_date,1,2)||'-'||substr(t.due_date,4,2)) BETWEEN (SELECT d30 FROM params) AND (SELECT today FROM params) THEN 1 ELSE 0 END) AS pending_30d FROM people pe LEFT JOIN tasks t ON t.assigned_to = pe.person_id GROUP BY pe.person_id, person_name ORDER BY completed_30d DESC, pending_30d DESC;
Q3 — Portfolio Status SummaryRequired
Business question: By project status, how many projects exist and what is the average task completion rate? Tells leadership where velocity is high vs. where projects are stalling.
Techniques: Nested CTEs · AVG(CASE WHEN) for completion rate · COALESCE · GROUP BY status · ORDER BY count
WITH task_rate AS ( SELECT project_id, AVG(CASE WHEN completed = 1 THEN 1.0 ELSE 0.0 END) AS completion_rate FROM tasks GROUP BY project_id ) SELECT p.status AS project_status, COUNT(*) AS project_count, ROUND(AVG(COALESCE(tr.completion_rate,0)) * 100.0, 1) AS avg_task_completion_pct FROM projects p LEFT JOIN task_rate tr ON tr.project_id = p.project_id GROUP BY p.status ORDER BY project_count DESC;
Q4 — Overdue Task Hot ListRequired
Business question: Which incomplete tasks have already passed their due date? Daily stand-up input — shows task, project, owner, and how overdue in one view for immediate action.
Techniques: CTE date anchor · 3-table JOIN (tasks, projects, people) · substr() date comparison · ORDER BY date ASC
WITH today AS (SELECT date('now') AS d) SELECT t.task_name, p.project_name, pe.first_name || ' ' || pe.last_name AS owner, t.due_date, date(substr(t.due_date,7,4)||'-'||substr(t.due_date,1,2)||'-'||substr(t.due_date,4,2)) AS due_iso FROM tasks t JOIN projects p ON p.project_id = t.project_id LEFT JOIN people pe ON pe.person_id = t.assigned_to WHERE t.completed = 0 AND t.due_date IS NOT NULL AND date(substr(t.due_date,7,4)||'-'||substr(t.due_date,1,2)||'-'||substr(t.due_date,4,2)) < (SELECT d FROM today) ORDER BY due_iso ASC, project_name, task_name;
Q5 — Above-Average Spend CategoriesRequired
Business question: Which expense categories have total spend above the cross-category average? Focuses cost-reduction attention on the highest-impact categories only.
Techniques: Two-level CTEs · self-referencing WHERE with scalar subquery · cross-join to compare against aggregate · ORDER BY DESC
WITH cat AS ( SELECT ec.category_name, ROUND(SUM(e.amount), 2) AS total_spend FROM expenses e LEFT JOIN expense_categories ec ON ec.expense_category_id = e.expense_category_id GROUP BY ec.category_name ), overall AS (SELECT AVG(total_spend) AS avg_spend FROM cat) SELECT c.category_name, c.total_spend FROM cat c, overall o WHERE c.total_spend > o.avg_spend ORDER BY c.total_spend DESC;
Q6 — Top 3 Projects by Task Throughput (90 days)Required
Business question: Which 3 projects completed the most tasks in the last 90 days? Highlights high-velocity teams for recognition or resource reinforcement.
Techniques: Triple CTE chain · ROW_NUMBER() window function · COALESCE for zero-task projects · filtered ranking with WHERE rn ≤ 3
WITH params AS (SELECT date('now','-90 day') AS d90), counts AS ( SELECT project_id, COUNT(*) AS completed_90d FROM tasks WHERE completed = 1 AND date(substr(completion_date,7,4)||'-'||substr(completion_date,1,2)||'-'||substr(completion_date,4,2)) >= (SELECT d90 FROM params) GROUP BY project_id ), ranked AS ( SELECT p.project_name, COALESCE(c.completed_90d,0) AS completed_90d, ROW_NUMBER() OVER (ORDER BY COALESCE(c.completed_90d,0) DESC, p.project_name) AS rn FROM projects p LEFT JOIN counts c ON c.project_id = p.project_id ) SELECT project_name, completed_90d FROM ranked WHERE rn <= 3 ORDER BY completed_90d DESC;
Q7 — Overloaded Team Members (Next 7 Days)Original
Business question (original): Who has more estimated hours due in the next 7 days than their weekly capacity? Proactive rebalancing before deadlines slip — not reactive after. Returns 0 rows in this dataset, which is itself a valid confirmation that no one is currently overloaded.
Techniques: Date window CTE · SUM of estimated_hours per person · JOIN against people.weekly_capacity_hours · WHERE overload condition · ORDER by severity
WITH due_next7 AS ( SELECT assigned_to AS person_id, SUM(COALESCE(estimated_hours, 0)) AS est_due_next7 FROM tasks WHERE completed = 0 AND assigned_to IS NOT NULL AND date(substr(due_date,7,4)||'-'||substr(due_date,1,2)||'-'||substr(due_date,4,2)) BETWEEN date('now') AND date('now','+7 day') GROUP BY assigned_to ) SELECT pe.first_name || ' ' || pe.last_name AS person_name, pe.weekly_capacity_hours, ROUND(COALESCE(d.est_due_next7, 0), 2) AS est_due_next7, ROUND(COALESCE(d.est_due_next7, 0) - pe.weekly_capacity_hours, 2) AS overload_hours FROM people pe LEFT JOIN due_next7 d ON d.person_id = pe.person_id WHERE COALESCE(d.est_due_next7, 0) > pe.weekly_capacity_hours ORDER BY overload_hours DESC, person_name;
Q8 — Client Profitability SnapshotOriginal
Business question (original): For each client, what is the total active project budget vs. total spend, and what is the variance %? Sorted by worst variance — surfaces unprofitable client relationships to renegotiate scope or fees. A CFO-level view that the required queries don't provide.
Techniques: Two parallel CTEs · triple LEFT JOIN (clients, proj, spend) · COALESCE throughout · NULLIF division guard · sorted by financial severity
WITH proj AS ( SELECT p.client_id, COUNT(*) AS active_projects, SUM(CASE WHEN p.is_active = 1 THEN p.budget ELSE 0 END) AS total_budget FROM projects p GROUP BY p.client_id ), spend AS ( SELECT p.client_id, SUM(e.amount) AS total_spend FROM projects p LEFT JOIN expenses e ON e.project_id = p.project_id GROUP BY p.client_id ) SELECT c.client_name, COALESCE(pr.active_projects, 0) AS active_projects, ROUND(COALESCE(pr.total_budget, 0), 2) AS total_budget, ROUND(COALESCE(sp.total_spend, 0), 2) AS total_spend, ROUND(COALESCE(sp.total_spend,0) - COALESCE(pr.total_budget,0), 2) AS variance_amount, ROUND(100.0 * COALESCE(sp.total_spend,0) / NULLIF(COALESCE(pr.total_budget,0),0),1) AS pct_of_budget FROM clients c LEFT JOIN proj pr ON pr.client_id = c.client_id LEFT JOIN spend sp ON sp.client_id = c.client_id ORDER BY variance_amount DESC, c.client_name;
06 — Reflection

What I Learned —
Honest Assessment

The most important lesson wasn't technical — it was architectural. Designing the ERD before writing a single line of SQL forced clarity about what the business actually needs to track vs. what's easy to store. Every foreign key encodes a business rule. Every constraint is a policy decision. The discipline of deciding cardinality on paper before touching code produces schemas that answer business questions — not just schemas that store data.

The import sequence revealed something deeper: the import order and the ERD dependency graph are the same artifact viewed differently. A schema with 9 tables has exactly one valid import order. Understanding that conceptually — before DBeaver enforces it with errors — is the difference between a database designer and someone following instructions.

Query 7 returning zero rows was initially surprising. Then it was the most instructive result in the project. A query that confirms no one is overloaded is as valuable as one that finds overloads — SQL is a verification mechanism, not just a retrieval engine. The real skill is trusting a well-written query even when the result is "nothing wrong."

AI Usage — What Helped vs. What Broke
Helped
Schema generation from ERD
Prompt: "Generate full SQLite schema from my ERD with foreign keys." Used to cross-check my design — then every line was manually reviewed against the ERD for correctness before execution.
Helped
Complex query structuring with CTEs
Prompt: "Help me structure a query that compares each person's upcoming task hours against their weekly capacity." Produced a useful CTE skeleton that I adapted with correct SQLite date parsing logic.
Fixed
MySQL syntax in SQLite context — AUTO_INCREMENT and VARCHAR(255)
AI repeatedly suggested MySQL-specific syntax. SQLite uses AUTOINCREMENT (no underscore) and TEXT (not VARCHAR). Caught by testing each CREATE TABLE in DBeaver before the next was written. Fixed across the entire schema.
Fixed
Wrong import order recommendation
AI suggested a random re-ordering when FK constraint failures appeared during import — which made the problem worse. The correct fix (strict parent-to-child order matching the ERD dependency graph) was discovered by reasoning from first principles, not from AI advice.
Fixed
Auto-column mapping in DBeaver
AI suggested allowing DBeaver to auto-map CSV columns during import. Auto-mapping silently mismatched assigned_person → assigned_to and similar field name variants. Fixed by manually verifying every source-to-target column mapping in the import wizard before starting each import.

"AI accelerated the work — but every AI suggestion was treated as a first draft, not a final answer. The skill is knowing which output to trust, which to fix, and which to reject entirely."
